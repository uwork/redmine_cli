# テスト方針

## 基本ルール

- テストは各ファイル末尾の `#[cfg(test)] mod tests { ... }` ブロック内に書くこと
- ネットワーク・ファイルシステムに依存しない純粋な単体テストを書くこと
- テスト名は「何をテストするか」を英語スネークケースで明示すること

## テスト対象と優先順位

1. **純粋ロジック** (最優先): 入出力が確定できるものから書くこと
   - `Config::require_url` / `require_api_key` のエラー/成功パス
   - `User::full_name` などのメソッド
   - TOML シリアライズ/デシリアライズの往復確認

2. **JSON 入出力** (優先): `serde_json` を使って API 型の正確さを検証すること
   - `#[serde(skip_serializing_if)]` が None フィールドを省略するか確認すること
   - `Default` 実装が空オブジェクトにシリアライズされるか確認すること

3. **初期化ロジック**: 副作用のない構築処理を検証すること
   - `RedmineClient::new` が URL の末尾スラッシュをトリムするか確認すること

## やらないこと

- 実際の HTTP 通信を伴うテストは書かないこと (モックなしでは不要)
- `Config::save` / `Config::load` のようなファイルシステムを変更するテストは書かないこと
- `std::env::set_var` を使う環境変数テストは書かないこと (並列実行時に競合するため)
